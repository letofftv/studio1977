# Статусы, уведомления и аудит

## 1. Статусы проекта

### 1.1 Верхнеуровневый статус
- draft
- active
- on_hold
- waiting_client
- completed
- archived
- cancelled

### 1.2 Этап проекта
- onboarding
- discovery
- strategy
- concept
- production
- review
- approval
- delivery
- support

Важно:
верхнеуровневый статус и этап не одно и то же.

## 2. Статусы согласования
- pending
- approved
- revision_requested
- cancelled
- expired

## 3. Статусы файлов
- uploaded
- processing
- available
- replaced
- archived
- blocked

## 4. Статусы синхронизации Bitrix24
- queued
- processing
- success
- failed
- retry_scheduled
- dead_letter

## 5. Когда шлются уведомления клиенту

- новый проект приглашен в кабинет
- смена клиентского статуса проекта
- запрос согласования
- новый документ
- новый файл, видимый клиенту
- новый публичный комментарий
- приближение контрольной даты
- менеджер ответил на запрос

Каналы:
- внутри кабинета
- email
- push web-app, если включено и реализовано

## 6. Когда шлются уведомления команде

- новая заявка
- новый бриф
- клиент загрузил файл
- клиент оставил комментарий
- клиент согласовал этап
- клиент запросил доработку
- ошибка синхронизации
- просроченный дедлайн
- проблемный статус проекта

## 7. Аудит

## 7.1 Что обязательно писать в audit_log
- вход пользователя
- смена роли
- создание проекта
- изменение статуса проекта
- загрузка и удаление файла
- скачивание критичного документа
- согласование этапа
- изменение связки с Bitrix24
- ручной запуск ресинка
- ошибки интеграции

## 7.2 Поля записи аудита
- actor
- action
- target_entity
- before
- after
- ip
- user_agent
- timestamp

## 8. Бизнес-правила

- клиентское действие согласования нельзя потерять без следа
- системный лог не должен быть редактируем пользователями
- у каждого уведомления должен быть deep-link
- прочтение уведомления не удаляет сам факт события из журнала

## 9. Acceptance criteria

- любое изменение статуса порождает системное событие
- если событие клиентское, оно видно клиенту в timeline
- если событие внутреннее, оно видно только команде
- сбой интеграции не ломает основной UX, а попадает в очередь ошибок
