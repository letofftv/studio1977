# Логика кабинетов: клиент и команда

## 1. Клиентский кабинет

## 1.1 Цель
Клиент должен открывать кабинет не как сложную CRM, а как понятное окно в свой проект.

## 1.2 Обязательные модули клиента
- Дашборд
- Мои проекты
- Карточка проекта
- Файлы
- Брифы
- Согласования
- Документы
- Комментарии
- Уведомления
- Профиль

## 1.3 Что клиент видит в карточке проекта
Верхний блок:
- название
- статус
- текущий этап
- дедлайн ближайшей точки
- менеджер проекта
- кнопка связи
- индикатор новых событий

Средний блок:
- таймлайн
- ближайшие действия
- что требуется от клиента сейчас
- последние файлы
- последние комментарии

Нижний блок:
- документы
- история проекта
- архив

## 1.4 Что нельзя делать кабинет слишком похожим на CRM
Нельзя показывать:
- внутренние названия воронок
- лишние технические статусы
- поля типа `UF_CRM_*`
- внутренние задачи команды
- внутренние обсуждения и проблемные пометки

## 2. Кабинет команды

## 2.1 Цель
Дать команде единый слой поверх CRM и файлов, чтобы проектом было удобно управлять.

## 2.2 Обязательные модули
- Дашборд
- Проекты
- Карточка проекта
- Лиды
- Клиенты
- Контент и кейсы
- Интеграции
- Аудит
- Настройки

## 2.3 Карточка проекта команды
Блоки:
- шапка и мета
- клиент
- CRM-связки
- состав команды
- этапы
- задачи
- файлы
- комментарии
- согласования
- документы
- лог синхронизации
- аудит

## 2.4 Режимы отображения
- канбан по статусам
- таблица
- карточки
- календарь по дедлайнам

## 3. Общие модули

### 3.1 Комментарии
Типы:
- client_public
- team_internal
- system
- approval_note

### 3.2 Файлы
Категории:
- contract
- invoice
- brief
- reference
- source
- result
- presentation
- photo_video
- act
- other

### 3.3 Документы
Типы:
- договор
- счет
- акт
- смета
- КП
- приложение
- NDA
- иное

## 4. Быстрые действия

### Для клиента
- загрузить файл
- посмотреть новый комментарий
- согласовать этап
- скачать документ
- написать менеджеру

### Для команды
- обновить этап
- создать согласование
- отправить клиенту комментарий
- привязать CRM сущность
- перезапустить sync
- отметить проблему

## 5. Системные индикаторы

Нужны на карточке проекта:
- sync ok / sync error
- есть непрочитанные комментарии
- есть ожидаемое согласование
- просрочен дедлайн
- ожидаются материалы от клиента
- документ требует действия

## 6. Mobile-first логика

Клиентский кабинет обязан удобно работать как web-app:
- нижняя навигация
- фиксированная кнопка быстрых действий
- свайп-карточки проекта допустимы
- большие touch-зоны
- базовые действия доступны одной рукой

## 7. Empty states

Для каждого пустого состояния нужны тексты и действия:
- нет проектов
- нет новых файлов
- нет уведомлений
- нет комментариев
- нет документов
- нет согласований

## 8. Design-system требования для кабинетов

- строгая иерархия типографики
- карточная логика
- единые статусы и бейджи
- четкие hover / active / disabled состояния
- skeleton loading
- toast уведомления
- светлая и темная тема опционально, если хватит ресурса
