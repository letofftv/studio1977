# Роли, права и пользовательские сценарии

## 1. Роли

### 1.1 Guest
Может:
- смотреть публичные страницы
- отправлять заявку
- заполнять бриф
- подписываться на новости
Не может:
- видеть кабинеты
- видеть проектные данные
- загружать материалы в закрытые зоны

### 1.2 Lead
Это еще не клиент, но уже есть запись в системе.
Может:
- получить magic-link или временный доступ к брифу
- дослать недостающие материалы
- подтвердить контактные данные

### 1.3 Client Owner
Главный представитель клиента.
Может:
- видеть все проекты своей компании
- приглашать коллег в кабинет
- смотреть документы
- согласовывать этапы
- комментировать
- загружать файлы
- менять профиль компании в рамках разрешенных полей

### 1.4 Client Member
Сотрудник со стороны клиента.
Может:
- видеть назначенные ему проекты
- оставлять комментарии
- загружать файлы
- участвовать в согласовании, если выдано право
Не может:
- управлять доступами компании
- видеть скрытые финансовые документы без отдельного права

### 1.5 Account Manager
Может:
- создавать проект
- назначать команду
- менять статусы проекта
- комментировать
- инициировать согласование
- видеть CRM-данные
- запускать ручную синхронизацию по проекту

### 1.6 Producer / PM
Может:
- вести таймлайн
- обновлять этапы
- заводить внутренние задачи
- управлять составом deliverables
- загружать материалы
- работать с согласованиями

### 1.7 Specialist
Может:
- видеть проекты, где он участвует
- работать с файлами и задачами
- писать внутренние комментарии
- менять статусы только в пределах разрешенной зоны

### 1.8 Admin
Полный доступ:
- роли
- доступы
- интеграции
- аудит
- справочники
- системные настройки
- сопоставление полей Bitrix24

## 3. Главные пользовательские сценарии

### 3.1 Новый посетитель -> заявка
1. Заходит на главную
2. Переходит в услугу или кейс
3. Читает процесс работы
4. Нажимает CTA
5. Отправляет короткую заявку
6. Получает сообщение об успешной отправке
7. Создается лид на сайте и в Bitrix24

### 3.2 Новый посетитель -> расширенный бриф
1. Переходит в `/brief`
2. Выбирает направление работ
3. Заполняет форму по шагам
4. Загружает референсы
5. Отправляет
6. Получает письмо / сообщение с подтверждением
7. В CRM создается лид или сделка в зависимости от сценария

### 3.3 Менеджер -> создает проект из лида
1. Открывает лид
2. Проверяет данные
3. Конвертирует в контакт / компанию / сделку
4. Создает проект на сайте
5. Привязывает проект к CRM сущностям
6. Приглашает клиента в кабинет

### 3.4 Клиент -> первый вход в кабинет
1. Получает письмо со ссылкой
2. Проходит установку пароля или magic-link авторизацию
3. Попадает на onboarding
4. Видит список проектов или один активный проект
5. Понимает, что сейчас в работе и что требуется от него

### 3.5 Клиент -> согласование этапа
1. Получает уведомление
2. Открывает проект
3. Переходит во вкладку `Согласования`
4. Смотрит описание этапа и материалы
5. Нажимает:
   - согласовать
   - отправить на доработку
   - оставить комментарий
6. В системе создается запись действия и уходит событие в CRM

### 3.6 Клиент -> загрузка материалов
1. Открывает проект
2. Идет в `Файлы`
3. Выбирает тип:
   - бренд-материалы
   - договорные документы
   - референсы
   - контент
   - исходники
4. Загружает файл
5. Файл получает метаданные, автора, версию и привязку к проекту

### 3.7 Команда -> обновление статуса проекта
1. PM открывает карточку проекта
2. Меняет текущий этап
3. Добавляет комментарий
4. Выставляет новую контрольную дату
5. Клиент получает уведомление, если изменение видимое для него

### 3.8 Админ -> устраняет ошибку синхронизации
1. Заходит в `Интеграции`
2. Видит ошибочную задачу в очереди
3. Открывает payload
4. Перезапускает обмен
5. Видит результат и запись в логе

## 4. Onboarding клиента

После первого входа клиенту показывается:
- приветственный экран
- кто его менеджер
- где смотреть статусы
- где загружать материалы
- где согласовывать этапы
- как включить web-app на телефоне

## 5. Критичные UX-правила

- любое необратимое действие требует подтверждения
- клиент не должен видеть внутренние комментарии
- клиент всегда видит только свои проекты
- у каждого изменения статуса есть автор и время
- на мобиле все ключевые действия должны помещаться в один экран без перегруза

## 6. Acceptance criteria по ролям

- у каждого API и каждого экрана есть role-check
- скрытие на фронте не считается защитой, доступ проверяется и на сервере
- скачивание файла тоже должно проверять права
- ссылки на документы не должны быть публичными без токена или авторизации
