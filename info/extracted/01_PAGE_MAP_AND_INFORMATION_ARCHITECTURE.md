# Карта страниц и информационная архитектура

## 1. Общая логика

Сайт делится на 3 слоя:

1. Публичная витрина
2. Клиентский кабинет
3. Внутренний кабинет команды

Навигация должна оставаться ясной даже при масштабировании. Пользователь всегда должен понимать:
- где он находится
- что можно сделать дальше
- к какому проекту относится текущий экран
- какие действия доступны именно ему

## 3. Публичная часть

### 3.1 Главная `/`
Цель:
- моментально показать характер студии
- объяснить, чем занимаетесь
- перевести пользователя в услугу, кейс или заявку

Состав:
- hero
- позиционирование
- направления
- избранные кейсы
- подход к работе
- этапы
- лица проекта
- форма контакта
- footer

### 3.2 О студии `/about`
Цель:
- показать философию, контекст и зрелость команды
- объяснить, почему студии можно доверять

Состав:
- манифест
- история
- принципы работы
- роли внутри команды
- подход к клиентам
- CTA

### 3.3 Услуги `/services`
Цель:
- дать обзор всех направлений
- провести пользователя на конкретную услугу

Каждая карточка услуги содержит:
- что входит
- для кого подходит
- какой результат
- пример кейса
- CTA на бриф / заявку

### 3.4 Страница услуги `/services/[slug]`
Обязательные секции:
- заголовок и суть
- задачи, которые решает услуга
- состав работ
- процесс по этапам
- сроки / ориентиры
- кейсы по теме
- FAQ
- форма заявки

### 3.5 Кейсы `/cases`
Фильтры:
- по направлению
- по индустрии
- по типу проекта
- по году
- по формату: маркетинг / event / digital / брендинг

### 3.6 Страница кейса `/cases/[slug]`
Состав:
- обложка
- контекст
- задача
- решение
- результат
- галерея
- стек / формат работ
- следующий кейс
- CTA на похожий проект

### 3.7 Процесс `/process`
Цель:
- снять тревогу клиента
- показать, как устроена работа

Состав:
- этапы
- точки контроля
- согласования
- роль клиента
- как устроена коммуникация

### 3.8 Форматы сотрудничества `/formats`
Состав:
- проектная работа
- пакетные решения
- продюсерское сопровождение
- разовые услуги
- долгосрочное партнерство

### 3.9 Журнал `/journal`
Цель:
- SEO
- экспертность
- медийность
- трафик на услуги и кейсы

Типы материалов:
- статьи
- заметки
- разборы
- backstage
- обзоры рынка

### 3.10 Бриф `/brief`
Должно быть 2 режима:
- короткая заявка
- расширенный бриф

## 4. Клиентский кабинет `/app`

## 4.1 Дашборд `/app`
Что видит клиент:
- активные проекты
- последние обновления
- ближайшие согласования
- новые файлы
- документы к подписанию
- уведомления
- быстрые действия

Быстрые действия:
- загрузить файл
- оставить комментарий
- открыть проект
- подтвердить этап
- скачать документ
- задать вопрос

## 4.2 Проекты `/app/projects`
Список карточек:
- название
- тип проекта
- статус
- текущий этап
- ответственный менеджер
- дедлайн ближайшей точки
- индикатор новых событий

## 4.3 Карточка проекта `/app/projects/[projectId]`
Вкладки:
- обзор
- таймлайн
- файлы
- брифы
- согласования
- документы
- комментарии

## 4.4 Уведомления `/app/notifications`
Типы:
- новый комментарий
- новый файл
- изменение статуса
- запрос согласования
- документ готов
- дедлайн приближается

## 4.5 Профиль `/app/profile`
Поля:
- имя
- телефон
- email
- компания
- должность
- способ связи
- настройки уведомлений

## 5. Внутренний кабинет `/studio`

## 5.1 Главный экран `/studio`
Содержимое:
- проекты в работе
- просроченные задачи
- последние лиды
- активность команды
- проблемные проекты
- лента событий

## 5.2 Проекты `/studio/projects`
Табличный и карточечный виды.
Фильтры:
- статус
- направление
- менеджер
- клиент
- дедлайн
- стадия

## 5.3 Карточка проекта `/studio/projects/[projectId]`
Блоки:
- шапка проекта
- клиент
- состав команды
- этапы
- внутренние задачи
- ссылки на Битрикс24 сущности
- документы
- файлы
- комментарии
- журнал действий

## 5.4 Лиды `/studio/leads`
Лента входящих:
- заявки с сайта
- брифы
- запросы контакта
- источники
- статус обработки
- привязка к CRM записи

## 5.5 Интеграции `/studio/integrations`
Служебный экран:
- статус Bitrix24
- последнее обновление синхронизации
- число ошибок
- очередь вебхуков
- ручной рефреш
- лог обмена

## 6. Навигационные принципы

- публичная часть имеет обычное верхнее меню
- кабинет клиента имеет нижнюю навигацию на мобиле и сайдбар на десктопе
- кабинет команды имеет сайдбар, breadcrumbs и быстрые фильтры
- на всех проектных страницах обязательно показывать контекст проекта в шапке
- важные действия должны быть доступны без глубокого проваливания

## 7. URL-правила

- все slug латиницей
- человекочитаемые адреса
- один канонический URL на страницу
- UTM не должны ломать маршрутизацию
- закрытые маршруты не индексируются

## 8. Что должен получить дизайнер по этому файлу

- sitemap
- набор шаблонов страниц
- понимание приоритетов блоков
- состав экранов для desktop и mobile
