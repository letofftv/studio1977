# ТЗ по интеграции сайта "Студия 1977" с Битрикс24

## 1. Общая задача

Нужно связать сайт, личные кабинеты и внутреннюю платформу студии с Битрикс24 так, чтобы:
- все заявки не терялись
- контакты и компании корректно создавались в CRM
- проекты были связаны со сделками или смарт-процессами
- команда видела актуальные статусы
- часть автоматизаций работала через Битрикс24
- клиентский опыт оставался внутри сайта

Битрикс24 в рамках этого ТЗ рассматривается как CRM с REST API, вебхуками и инструментами для интеграции, что подходит для такой архитектурной модели.

## 2. Основной принцип интеграции

Битрикс24 в этом проекте используется как CRM и автоматизационное ядро, но не как основной пользовательский интерфейс клиента.

### Значит:
- публичный сайт живет отдельно
- личные кабинеты живут отдельно
- backend сайта синхронизируется с Битрикс24
- критичные CRM-сущности хранят связь с сущностями сайта

## 3. Почему не делать все напрямую через формы Битрикс24

Потому что проекту нужны:
- собственный визуал
- собственные кабинеты
- web-app формат
- гибкие проектные сущности
- брендированный клиентский опыт
- своя логика прав доступа и интерфейсов

## 4. Что должно синхронизироваться

## 4.1 Из сайта в Битрикс24
- лиды
- контакты
- компании
- сделки
- задачи при необходимости
- комментарии для внутренней работы при выбранной модели
- служебные поля проекта
- источники, utm-метки, referrer

## 4.2 Из Битрикс24 на сайт
- статусы CRM-сущностей
- ответственные менеджеры
- информация о связанных клиентах
- важные триггерные события
- этапы сделки или этапы смарт-процесса
- внутренние поля, которые участвуют в логике кабинета

## 5. Рекомендуемая модель сущностей

### На стороне сайта
- user
- client_profile
- project
- task
- file
- approval
- document
- notification

### На стороне Битрикс24
- lead
- contact
- company
- deal
- task
- custom fields
- smart-process, если проектная логика будет расширяться

REST API и вебхуки используются здесь как базовый способ обмена данными между сайтом и CRM.

## 6. Карта связей

Нужно хранить таблицу соответствий:
- `site_user_id <-> bitrix_contact_id`
- `site_company_id <-> bitrix_company_id`
- `site_project_id <-> bitrix_deal_id` или `smart_process_id`
- `site_task_id <-> bitrix_task_id`, если включена двусторонняя синхронизация задач
- `site_request_id <-> bitrix_lead_id`

## 7. Как использовать Битрикс24 в этом проекте

## 7.1 Для лидогенерации
Все формы сайта должны отправлять данные в backend, а backend уже создает нужную сущность в Битрикс24.

### Обязательные данные по заявке
- имя
- телефон
- email
- услуга
- комментарий
- страница-источник
- utm_source
- utm_medium
- utm_campaign
- utm_content
- utm_term
- referrer
- дата и время

## 7.2 Для проектной логики
После квалификации заявки в CRM:
- создается сделка
- к сделке привязывается контакт и при необходимости компания
- проект сайта получает ссылку на CRM-сущность
- менеджер получает доступ к связке в кабинете

## 7.3 Для внутренней работы
Можно использовать:
- сделки для проектного уровня
- задачи Битрикс24 для части операционной работы
- смарт-процессы, если проектная модель усложнится

## 8. Способ интеграции

Для базового и среднего уровня сложности рекомендуется модель через backend сайта + входящие / исходящие вебхуки Битрикс24.

Для базовой интеграции в этом проекте принимается модель: backend сайта работает с CRM через входящие и исходящие вебхуки и отдельные точки интеграции.

### Почему так удобнее
- проще контролировать логику
- можно нормализовать данные
- можно вести лог ошибок
- можно повторять синхронизацию при сбоях
- можно скрыть внутренние ключи и не светить их на клиенте

## 9. Безопасность интеграции

### Обязательно
- не дергать Bitrix24 напрямую с frontend
- хранить webhook URL и секреты только на backend
- логировать ошибки синхронизации
- иметь retry-механику
- иметь ручной re-sync для администратора
- ограничить права webhook только нужными сущностями

Webhook-ключи и связанные секреты считаются чувствительными данными и должны храниться только на серверной стороне.

## 10. Вариант модели данных в Битрикс24

### Лиды
Используются для первичного входящего потока с сайта.

### Контакты
Создаются после первичной заявки или после квалификации.

### Компании
Создаются для юридических лиц и корпоративных клиентов.

### Сделки
Используются как основная сущность проектной работы на уровне продаж и сопровождения.

### Смарт-процессы
Можно использовать, если потребуется отдельная сущность "Проект", "Согласование" или "Производственный цикл".

## 11. Какие поля нужно предусмотреть в Битрикс24

### Для лида / сделки
- тип услуги
- формат проекта
- источник заявки
- страница входа
- utm-набор
- referrer
- ID проекта на сайте
- статус клиентского кабинета
- дата старта проекта
- планируемый дедлайн
- ссылка на проект в кабинете
- ответственный менеджер

### Для контакта
- имя
- телефон
- email
- роль
- связанный пользователь сайта
- компания
- комментарии

## 12. Автоматизации

Минимум нужно настроить:
- новая заявка с сайта -> создание лида
- лид квалифицирован -> создание сделки
- сделка создана -> создание проекта в системе или привязка к нему
- клиенту отправлено приглашение в кабинет
- смена стадии сделки -> обновление статуса проекта на сайте
- отправлено согласование -> уведомление менеджеру и клиенту
- согласование пройдено / отклонено -> обновление статуса

## 13. Ошибки и отказоустойчивость

Нужно предусмотреть:
- очередь или буфер запросов к Битрикс24
- лог критичных ошибок
- ручной повтор отправки
- уведомление администратора о сбоях интеграции
- защиту от дублирования сущностей

## 14. Что должен сделать интегратор

- создать нужные пользовательские поля в Битрикс24
- настроить вебхуки или приложение по выбранной модели
- дать карту сущностей и полей
- описать сценарии автоматизаций
- реализовать обмен данными через backend
- настроить логику дублирования и поиска контактов
- протестировать end-to-end сценарии

## 15. Обязательные тесты интеграции

- заявка с сайта создает сущность в CRM
- повторная заявка того же клиента не создает хаос в дублях
- проект в кабинете связан с нужной CRM-сущностью
- изменение стадии в CRM отражается в кабинете
- ошибка CRM не ломает пользовательский сценарий на сайте
- все webhook-ключи и секреты недоступны на клиенте

## 16. Критерии приемки

Интеграция считается принятой, если:
- данные стабильно ходят в обе стороны там, где это требуется
- менеджер видит связку сайта и CRM
- клиент не сталкивается с CRM-интерфейсом напрямую
- автоматизации работают без ручного хаоса
- есть документация по полям, сценариям и webhook-точкам
