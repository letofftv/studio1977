# ТЗ по web-app и PWA для сайта "Студия 1977"

## 1. Общая задача

Сайт должен работать не только как обычный сайт, но и как web-app, который можно установить на телефон и использовать как быстрый рабочий инструмент.

Это особенно важно для:
- клиента, который хочет быстро открыть проект, файлы, согласования и документы
- команды студии, которая работает с проектами на ходу
- менеджера, которому важно быстро реагировать на статусы и события

## 2. Что понимается под web-app в этом проекте

Под web-app здесь понимается:
- адаптивный сайт с приложенческим сценарием использования
- возможность установки на главный экран
- собственная иконка и имя
- полноэкранный или близкий к нему режим
- быстрый вход
- удобная мобильная навигация
- сохранение пользовательской сессии по безопасной модели

## 3. Что должно быть реализовано

### 3.1 PWA-база
- web manifest
- иконки нужных размеров
- корректное имя приложения
- цвет темы
- стартовый экран
- корректный display mode
- service worker
- offline fallback для допустимых сценариев

### 3.2 Ускорение повторных посещений
- кэширование статических ресурсов
- кэширование shell-части интерфейса
- быстрый повторный вход
- предсказуемая работа после установки

### 3.3 Приложенческий UX
- мобильная навигация
- быстрые действия на дашборде
- компактные карточки проектов
- удобный просмотр статусов
- удобная загрузка файлов
- простое согласование с мобильного
- понятная работа с уведомлениями

## 4. Что важно помнить

PWA не должно быть просто "сайтом, который можно установить".  
Оно должно ощущаться как удобный инструмент.

Значит:
- мобильный сценарий проектируется отдельно
- главные действия на 1-2 касания
- никакой перегрузки лишними блоками
- вся критичная информация наверху

## 5. Обязательные мобильные сценарии

### Для клиента
- открыть приложение
- быстро войти
- увидеть проекты
- открыть проект
- понять, что нужно сделать
- загрузить файл с телефона
- согласовать этап
- открыть документ

### Для команды
- открыть приложение
- увидеть активные проекты
- открыть карточку
- проверить статус
- отправить согласование
- посмотреть уведомление
- быстро перейти в нужный проект

## 6. Навигация

Нужно продумать мобильную навигацию так, чтобы ею было реально удобно пользоваться.

### Допустимые модели
- нижняя навигация для главных разделов
- верхняя панель + drawer
- гибридная схема

### Что важно
- не прятать важные действия слишком глубоко
- оставить понятный путь назад
- выделять текущий раздел
- учитывать сценарии с одной рукой

## 7. Авторизация в web-app

Нужно:
- продумать безопасное хранение сессии
- корректно обрабатывать истечение сессии
- не мучить пользователя постоянными повторными входами без причины
- дать понятный сценарий восстановления доступа

## 8. Offline и кэш

### На первом этапе
Достаточно:
- offline fallback экран
- кэш оболочки
- кэш части нединамичных ресурсов

### Что не нужно обещать без отдельной реализации
- полноценную offline-работу с изменением проектных данных
- гарантированную синхронизацию оффлайн-редактирования

## 9. Push и уведомления

### На MVP
Достаточно:
- уведомления внутри кабинета
- email по критичным событиям

### На втором этапе
Можно рассмотреть:
- web push
- более глубокую мобильную механику уведомлений

Важно: логика уведомлений не должна зависеть только от одной технологии доставки.

## 10. Дизайн web-app версии

Нужно:
- адаптировать типографику под мобильный сценарий
- сократить визуальный шум
- укрупнить кликабельные области
- продумать sticky-панели для быстрых действий
- аккуратно работать с модальными окнами на мобильном

## 11. Технические требования

- manifest должен собираться корректно
- service worker не должен ломать обновления
- нужно предусмотреть стратегию обновления кэша
- приложение должно корректно обновляться после релиза
- нельзя кэшировать чувствительные данные без продуманной логики безопасности

## 12. Что нужно протестировать

- установка на главный экран
- запуск из иконки
- повторный вход
- работа на мобильных браузерах
- открытие кабинета
- загрузка файлов
- навигация между разделами
- поведение после обновления релиза
- offline fallback

## 13. Критерии приемки PWA/web-app

Web-app часть считается готовой, если:
- сайт устанавливается как приложение
- имеет нормальную иконку и имя
- быстро открывается повторно
- клиенту удобно использовать кабинет с телефона
- команда может решать базовые рабочие задачи на мобильном
- обновления не ломают установленную версию
