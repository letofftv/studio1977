# Общая архитектура продукта: сайт "Студия 1977"

## 1. Суть проекта

Сайт "Студия 1977" должен быть не просто презентацией агентства, а цифровой рабочей средой, где соединяются:

- брендинг и визуальная подача студии
- продажа услуг и захват заявок
- демонстрация кейсов и экспертизы
- коммуникация с клиентами
- внутренняя работа команды
- проектное сопровождение
- интеграция с Битрикс24
- web-app формат для быстрого доступа с телефона

## 2. Главные задачи сайта

### 2.1 Имиджевая
Показать студию как сильный, современный, нестандартный, визуально взрослый проект с авторским подходом.

### 2.2 Коммерческая
Приводить заявки на:
- маркетинговые услуги
- event-направление
- продюсирование
- упаковку брендов
- digital-разработку
- отдельные проектные решения

### 2.3 Операционная
Дать клиенту и команде удобную точку входа в проект без постоянной переписки по разным мессенджерам и таблицам.

### 2.4 Сервисная
Сделать личный кабинет, где клиент может:
- видеть статус проекта
- отправлять материалы
- согласовывать этапы
- оставлять комментарии
- смотреть документы и счета
- работать с брифами

## 3. Типы пользователей

### 3.1 Гость сайта
Человек, который впервые попал на сайт.

Цели:
- понять, кто вы
- оценить уровень студии
- посмотреть кейсы
- понять, можно ли вам доверять
- оставить заявку

### 3.2 Потенциальный клиент
Человек, который уже интересуется конкретной услугой.

Цели:
- изучить процессы
- посмотреть кейсы по направлению
- понять порядок работы
- отправить бриф или заявку
- записаться на созвон

### 3.3 Действующий клиент
Человек, у которого уже идет проект.

Цели:
- зайти в личный кабинет
- увидеть статус проекта
- загрузить материалы
- согласовать этап
- посмотреть документы
- понять следующий шаг

### 3.4 Команда студии
Внутренние сотрудники и подрядчики.

Цели:
- видеть проекты и статусы
- получать задачи
- отмечать прогресс
- вести внутренние комментарии
- работать с файлами и дедлайнами
- синхронизироваться с Битрикс24

### 3.5 Администратор / менеджер проекта
Цели:
- управлять доступами
- создавать проекты
- назначать роли
- отслеживать активность
- модерировать клиентские материалы
- контролировать синхронизацию с CRM

## 6. Принципы визуальной подачи

- визуально взрослый сайт
- не шаблонный "корпоратив"
- акцент на композицию, типографику, ритм, арт-дирекшн
- современный digital luxury без лишней вычурности
- много воздуха
- сильные контрасты
- продуманная анимация
- ощущение премиального, но живого продукта
- сайт должен выглядеть как сайт студии, которая умеет в вкус, процесс и результат

## 7. Принципы функционала

### 7.1 Удобство
Пользователь не должен думать, куда нажать. Навигация ясная, сценарии короткие, все важное доступно в 1-3 клика.

### 7.2 Скорость
Сайт не должен ощущаться тяжелым даже при богатом визуале.

### 7.3 Масштабируемость
Архитектура должна позволять добавлять:
- новые услуги
- новые кейсы
- новые роли пользователей
- новые статусы и сущности
- дополнительные кабинеты
- расширения через API

### 7.4 Контроль доступа
Нужна нормальная ролевая модель:
- гость
- клиент
- менеджер
- участник команды
- администратор

## 9. Предпочтительная архитектурная модель

### 9.1 Frontend
Отдельное кастомное frontend-приложение сайта и кабинетов.

### 9.2 Backend
Отдельный backend с API, бизнес-логикой, авторизацией, файлами, синхронизацией и журналом действий.

### 9.3 CRM
Битрикс24 используется как CRM-ядро для заявок, сделок, контактов, компаний, задач и автоматизаций.

### 9.4 Почему не делать все только внутри Битрикс24
Потому что:
- публичный сайт должен быть визуально сильнее типового CRM-интерфейса
- клиентский опыт должен быть единым и брендированным
- web-app логика удобнее реализуется на собственной платформе
- нужны гибкие сценарии кабинетов, которых обычно мало в типовой CRM-обвязке

## 11. Требования к безопасности

- авторизация по email + пароль, либо email + одноразовый код
- защита закрытых маршрутов
- раздельные роли и права доступа
- журнал входов и ключевых действий
- защита от прямого доступа к файлам без прав
- защита форм от спама
- валидация данных на frontend и backend
- резервное копирование критичных данных

## 12. Нефункциональные требования

- адаптивность от мобильного до десктопа
- быстрая первая загрузка
- SEO-дружелюбная публичная часть
- удобная CMS-логика для редактирования контента
- расширяемость без полного переписывания системы
- доступность базового уровня
- корректная работа как web-app

- продает

- организует процессы
- упрощает работу с клиентом
- дает студии лицо и систему одновременно
