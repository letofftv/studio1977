# ТЗ по личным кабинетам: кабинет студии и кабинет клиента

## 1. Общая задача

На сайте должны быть реализованы два типа кабинетов:

1. Личный кабинет клиента
2. Внутренний кабинет студии

Они должны работать в одной системе, но иметь разную логику, роли и уровни доступа.

## 2. Зачем нужны кабинеты

### Кабинет клиента нужен для того, чтобы
- клиент видел статус проекта
- понимал, что происходит сейчас
- видел следующий шаг
- быстро загружал материалы
- согласовывал этапы
- находил документы
- не терялся в переписке

### Кабинет студии нужен для того, чтобы
- команда вела проекты в одном месте
- видела сроки и статусы
- работала с файлами
- следила за согласованиями
- управляла клиентским доступом
- была связка с CRM

## 3. Кабинет клиента

## 3.1 Основные разделы
- Дашборд
- Мои проекты
- Карточка проекта
- Файлы
- Согласования
- Документы
- Уведомления
- Профиль

## 3.2 Дашборд клиента
На дашборде нужно показать:
- активные проекты
- текущий статус по каждому
- ближайшее ожидаемое действие
- последние уведомления
- последние согласования
- быстрые действия

### Быстрые действия
- загрузить файл
- открыть проект
- посмотреть документы
- ответить на согласование
- написать комментарий

## 3.3 Список проектов клиента
Нужно показывать:
- название проекта
- тип услуги
- статус
- текущий этап
- дедлайн ближайшего шага
- ответственного менеджера
- кнопку "открыть"

## 3.4 Карточка проекта клиента
Это ключевой экран.

На карточке проекта должны быть:
- название проекта
- статус
- этап
- прогресс
- краткое описание
- команда проекта
- таймлайн этапов
- блок "что нужно от клиента сейчас"
- блок файлов
- блок согласований
- блок документов
- история действий

### Смысл карточки проекта
Клиент в любой момент должен понимать:
- где мы сейчас
- что уже сделано
- что нужно от него
- какой следующий шаг

## 3.5 Файлы клиента
Нужно дать возможность:
- загружать файлы
- видеть загруженные файлы
- скачивать доступные файлы
- понимать, к чему относится файл
- видеть дату загрузки и автора

### Фильтры
- по проекту
- по типу файла
- по дате
- по этапу

## 3.6 Согласования
Клиент должен иметь простой и очень понятный интерфейс согласования.

### На экране согласования должно быть
- что отправлено
- комментарий от команды
- прикрепленный файл / ссылка / превью
- кнопка "согласовать"
- кнопка "отправить на доработку"
- поле комментария
- история предыдущих согласований

## 3.7 Документы
Раздел должен включать:
- договоры
- счета
- акты
- другие проектные документы

Для каждого документа:
- название
- тип
- дата
- версия
- статус
- скачать

## 3.8 Уведомления
Уведомления нужны по событиям:
- новый этап
- новый запрос на согласование
- новый документ
- новый комментарий
- смена статуса проекта
- дедлайн или новое действие со стороны клиента

## 4. Внутренний кабинет студии

## 4.1 Основные разделы
- Дашборд
- Проекты
- Карточка проекта
- Задачи
- Клиенты
- Файлы
- Согласования
- Календарь
- Уведомления
- Команда
- Настройки
- Логи

## 4.2 Дашборд студии
На дашборде команды должны быть:
- активные проекты
- проекты с риском по срокам
- новые заявки
- последние согласования
- задачи на сегодня / неделю
- уведомления по системе
- быстрые действия

## 4.3 Проекты
Нужны:
- список
- фильтрация
- сортировка
- поиск
- быстрый просмотр статуса
- ответственный менеджер
- состав команды
- связанный клиент
- связка с CRM

## 4.4 Карточка проекта студии
Должна содержать:
- основную информацию
- статус
- этапы
- задачи
- состав команды
- клиента
- файлы
- документы
- согласования
- внутренние комментарии
- ленту активности
- ссылку на связанную сущность в Битрикс24

## 4.5 Задачи
Задачи можно хранить:
- внутри сайта
- либо в синхронизации с Битрикс24
- либо в гибридной логике, если это удобно проекту

Минимально нужно:
- список задач
- срок
- ответственный
- статус
- приоритет
- фильтры

## 4.6 Клиенты
Раздел нужен для:
- просмотра клиентов
- просмотра связанных проектов
- просмотра активности
- открытия карточки клиента
- перехода к CRM-сущности

## 4.7 Логи и история действий
Команда должна видеть:
- кто что изменил
- когда это произошло
- по какому проекту
- по какому объекту

## 5. Ролевая модель

### Клиент
- видит только свое
- не видит внутренние комментарии
- не видит внутренние служебные поля

### Участник команды
- видит свои назначенные проекты
- может работать с задачами, файлами и комментариями по проекту

### Менеджер
- управляет проектами, клиентским доступом, согласованиями и статусами

### Администратор
- имеет полный доступ

## 6. UX-принципы для кабинетов

- интерфейс должен быть спокойнее публичной части
- пользователь должен быстро ориентироваться
- важные действия должны быть на виду
- не должно быть перегруза мелкими блоками
- таблицы, карточки и списки должны быть читабельными
- мобильный сценарий должен быть полноценным

## 7. Мобильное использование

Так как кабинет должен работать и как web-app, мобильный сценарий обязателен.

Нужно продумать:
- нижнюю навигацию или удобный мобильный drawer
- компактный дашборд
- удобную загрузку файлов с телефона
- быстрый просмотр статусов
- простое согласование с мобильного
- удобный вход и повторный вход

## 8. Обязательные пользовательские сценарии

### Для клиента
- войти
- увидеть свои проекты
- открыть проект
- понять текущий статус
- загрузить файл
- согласовать или вернуть этап
- скачать документ

### Для команды
- войти
- открыть проект
- обновить статус
- отправить на согласование
- загрузить файл
- посмотреть историю действий
- проверить связку с CRM

## 9. Критерии приемки по кабинетам

Кабинеты считаются готовыми, если:
- у клиента нет путаницы, что происходит в проекте
- у команды есть понятная рабочая среда
- роли соблюдаются
- данные не смешиваются между клиентами
- мобильная версия реально удобна
- кабинет ощущается как часть бренда, а не как случайная админка
