# ТЗ для backend-разработки и платформенной части: сайт "Студия 1977"

## 1. Общая задача

Нужно разработать backend-платформу для сайта "Студия 1977", которая будет обслуживать:
- публичные заявки
- авторизацию
- личный кабинет клиента
- внутренний кабинет команды
- хранение проектных данных
- хранение файлов
- согласования
- уведомления
- интеграцию с Битрикс24
- web-app сценарии

## 2. Общая модель

Backend должен быть отдельным прикладным слоем, а не набором случайных обработчиков форм.

Он отвечает за:
- бизнес-логику
- доступы
- данные кабинетов
- API
- интеграции
- файловое хранилище
- журнал действий
- синхронизацию с CRM

## 3. Что должен уметь backend

### 3.1 Работа с пользователями
- регистрация пользователя через приглашение или после создания менеджером
- вход
- восстановление доступа
- смена пароля
- подтверждение email / одноразового кода при выбранной модели
- управление ролями
- блокировка пользователя
- аудит входов

### 3.2 Работа с проектами
- создание проекта
- редактирование проекта
- назначение клиента
- назначение команды
- смена статусов
- этапы проекта
- дедлайны
- история изменений
- привязка к сущности в Битрикс24

### 3.3 Работа с задачами
- создание
- редактирование
- назначение исполнителя
- статусы
- сроки
- комментарии
- чек-листы
- связь с проектом
- при необходимости связь с задачами в Битрикс24

### 3.4 Работа с файлами
- загрузка
- хранение
- версионирование
- назначение прав доступа
- скачивание
- удаление
- маркировка по типу
- связь с проектом, этапом, задачей, согласованием

### 3.5 Работа с согласованиями
- отправка этапа на согласование
- прием комментария от клиента
- статус согласовано / на доработку
- хранение истории версий
- уведомления участникам

### 3.6 Работа с документами
- список документов по проекту
- видимость по ролям
- история загрузок
- метки статуса
- версия документа

### 3.7 Уведомления
- системные уведомления
- уведомления о смене статуса
- уведомления о новом файле
- уведомления о новом комментарии
- уведомления о запросе на согласование
- уведомления о результате согласования

## 4. Роли и права

Минимальный набор ролей:
- admin
- manager
- team_member
- client

### Базовые правила
- клиент видит только свои проекты и связанные сущности
- team_member видит только те проекты, куда он назначен, если не задано иное
- manager видит свои и назначенные ему проекты
- admin видит все

## 5. API

Нужен нормальный API-слой с понятной структурой.

### Пример доменов API
- auth
- users
- projects
- tasks
- files
- approvals
- documents
- notifications
- profile
- bitrix

### Базовые требования
- единый формат ответов

- валидация входящих данных
- пагинация для списков
- фильтрация
- сортировка

## 7. Журнал действий

Обязателен activity log.

Нужно логировать:
- вход
- выход
- создание проекта
- смену статуса проекта
- загрузку файла
- удаление файла
- отправку на согласование
- ответ по согласованию
- изменение роли
- критичные ошибки интеграции

## 8. Работа с файлами

### Требования
- безопасная загрузка
- ограничение типов и размера файлов
- защита прямых ссылок
- метаданные файлов
- привязка к пользователю и проекту
- удаление только по правам
- подписанные URL или аналогичный защищенный механизм, если используется внешнее хранилище

## 9. Интеграционная логика

Backend должен уметь:
- принимать заявки с сайта
- создавать сущности в Битрикс24
- обновлять сущности в Битрикс24
- принимать события из Битрикс24
- синхронизировать статусы
- хранить карту связей между объектами сайта и CRM

## 10. Уведомления

### На первом этапе достаточно
- уведомления внутри кабинета
- email-уведомления о ключевых событиях

### На втором этапе можно добавить
- push для PWA
- интеграции с другими каналами

## 11. Безопасность

### Обязательно
- хеширование паролей
- rate limiting
- CSRF / XSS / basic security controls по выбранной архитектуре
- защищенные cookies или другой надежный механизм сессий
- проверка прав доступа на backend, а не только на frontend
- безопасное хранение ключей интеграций
- разделение dev/stage/prod окружений

## 12. Надежность

- логирование ошибок
- мониторинг ключевых сбоев
- retry там, где это оправдано для интеграций
- очередь или отложенная обработка для тяжелых задач и внешних запросов
- резервное копирование

## 13. Нужные backend-сценарии MVP

- заявка с сайта -> CRM
- создание клиента в системе
- авторизация пользователя
- просмотр клиентом своего проекта
- загрузка файла
- согласование этапа
- просмотр документов
- просмотр менеджером и командой списка проектов
- обновление статуса
- логирование действий

## 14. Админ-функции

- управление пользователями
- назначение ролей
- управление проектами
- просмотр логов
- повторная синхронизация с CRM
- ручной запуск отдельных интеграционных сценариев
- настройка справочников статусов

## 15. Что backend должен передать на запуск

- схема БД
- описание API
- env-example
- инструкция по развертыванию
- описание ролей и доступов
- инструкция по резервному копированию
- описание интеграции с Битрикс24
- список cron / jobs / очередей, если используются

## 16. Критерии приемки backend

Работа считается принятой, если:
- закрытая часть работает стабильно
- роли и доступы соблюдаются
- файл нельзя получить без прав
- статусы и сущности сохраняются корректно
- интеграция с Битрикс24 работает без хаоса
- логируются ключевые действия
- есть документация по API и запуску
