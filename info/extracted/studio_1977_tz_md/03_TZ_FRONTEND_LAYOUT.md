# ТЗ для верстальщика и frontend-разработки: сайт "Студия 1977"

## 1. Общая задача

Нужно сверстать и собрать frontend сайта "Студия 1977", включая:
- публичную часть
- страницу авторизации
- личный кабинет клиента
- внутренний кабинет команды
- web-app поведение
- связку с backend и Битрикс24 через API

Frontend должен быть:
- быстрым
- чистым
- масштабируемым
- адаптивным
- SEO-дружелюбным в публичной части
- удобным в закрытой части

## 2. Предпочтительный подход

### 2.1 Общий принцип
Не делать набор разрозненных шаблонов.  
Нужна компонентная система с едиными правилами стилей, состояний и повторного использования.

### 2.2 Рекомендуемая модель
- публичная часть и кабинеты работают в рамках одной системы
- общие токены дизайна
- единый UI-kit в коде
- разделение публичных и закрытых layout
- API-first подход

## 3. Требования к верстке

### 3.1 Семантика
- корректная HTML-структура
- логичные heading levels
- доступные формы
- корректные label, aria-атрибуты там, где нужно
- семантические секции, навигация, footer, main, aside

### 3.2 Адаптивность
Нужно поддержать:
- mobile
- tablet
- desktop
- широкие экраны

Контент и композиция не должны ломаться между брейкпоинтами.

### 3.3 Pixel-perfect
Нужна аккуратная реализация дизайн-макетов:
- отступы
- типографика
- размеры
- поведение состояний
- ритм блоков
- hover и transition

### 3.4 Аккуратность
- никаких случайных сдвигов
- никаких "почти похоже"
- никаких нестабильных костылей
- никаких захардкоженных значений там, где нужна система

## 4. Технические требования к frontend

## 4.1 Архитектура
Нужно:
- разделить layout публичной части и кабинетов
- вынести общие UI-компоненты
- вынести формы в переиспользуемые модули
- хранить дизайн-токены централизованно

- `api`

## 6.1 Личный кабинет клиента

- защищенный вход
- личный дашборд
- список проектов
- карточку проекта
- этапы

- уведомления
- профиль

## 6.2 Внутренний кабинет команды
Нужно реализовать:
- список проектов
- карточки проектов
- статусы
- задачи
- список клиентов
- файлы
- комментарии
- настройки
- логи действий

## 7. Формы

Нужно поддержать такие формы:
- короткая заявка
- полная заявка
- бриф
- вход
- восстановление доступа
- загрузка файлов
- комментарии
- согласование этапа
- редактирование профиля

### Требования к формам
- валидация на клиенте
- понятные ошибки
- маска телефона при необходимости
- защита от двойной отправки
- success-state
- интеграция с аналитикой и CRM
- сохранение utm и referrer, если они есть

## 8. Авторизация и доступ

Frontend должен уметь:
- показывать публичные маршруты без авторизации
- защищать маршруты кабинетов
- разделять доступ клиента и команды
- корректно обрабатывать истекшую сессию
- показывать понятный сценарий повторного входа

## 9. Работа с файлами

Нужно реализовать:
- drag&drop загрузку
- множественную загрузку
- отображение прогресса
- список загруженных файлов
- иконки по типам файлов
- удаление файла при наличии прав
- предпросмотр там, где возможно
- корректные ошибки загрузки

## 10. Анимации

Анимации реализуются только там, где они усиливают подачу и не вредят скорости.

Нужно:
- отдельно вынести motion-слой
- избегать анимаций, которые мешают SSR/SEO
- не ломать взаимодействие на мобильных
- уважать reduced motion preferences

## 11. Производительность

Frontend должен быть оптимизирован по следующим пунктам:
- lazy load для тяжелых блоков
- код-сплиттинг
- оптимизация изображений
- аккуратная работа со шрифтами
- ограничение лишнего JS на первом экране
- минимизация re-render
- skeleton loading там, где это оправдано

## 12. SEO и индексация

Для публичной части:
- уникальные meta title / description
- Open Graph
- корректная canonical-логика
- sitemap
- robots
- schema markup там, где нужно
- серверная или гибридная генерация важных страниц

Для кабинетов:
- закрытие от индексации

## 13. Логирование и обработка ошибок

Frontend должен:
- иметь централизованную обработку ошибок API
- показывать человеку понятный текст ошибки
- логировать критические frontend-ошибки
- не раскрывать системные детали пользователю

## 14. Требования к коду

- чистая структура
- читаемые имена
- без копипасты в больших объемах
- без магических чисел
- без хаотичных инлайн-стилей
- повторно используемые стили и компоненты
- строгая типизация, если стек это поддерживает
- комментарии только там, где реально нужно пояснение

## 17. Критерии приемки frontend-части

Работа считается принятой, если:
- все макеты реализованы корректно
- адаптив работает стабильно
- формы отправляются корректно
- кабинеты работают по ролям
- ошибки обрабатываются нормально
- сайт можно поставить на телефон как web-app
- frontend готов к интеграции или уже связан с backend/API
